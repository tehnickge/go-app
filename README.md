## goBack

Проект на стыке backend + frontend: backend уже построен вокруг понятной архитектуры и готов к развитию, а API/бизнес-слой и UI — в активной подготовке. Сейчас это не “готовый продукт”, но это хороший, расширяемый каркас: добавлять фичи будет быстро, потому что слои и ответственность уже разложены.

### Цель

Сделать сервис с управлением пользователями, группами и ролями (и связями между ними), где:
- доменные модели отделены от доступа к данным;
- бизнес-логика будет жить в отдельном слое;
- HTTP-контракты отделены от деталей БД;
- миграции и инициализация схемы встроены в старт приложения.

## Что уже сделано (и почему это выгодно)

1. Архитектурные слои заложены с самого начала.
   - `internal/domain` — модели (включая join-таблицы связей).
   - `internal/repository` — работа с БД через `gorm`.
   - `internal/usecase` и `internal/delivery/http` — подготовлены под бизнес-логику и HTTP-контракты, но пока не заполнены.
   - `pkg/db` — изолирует создание подключения к Postgres.
   Плюс: когда проект перейдёт в режим “фичи”, не придётся перепиливать структуру — добавления будут ложиться в правильный слой.

2. Миграции и начальная схема встроены в запуск.
   - `internal/migrations/migrate.go` создаёт extension `uuid-ossp`;
   - создаёт ENUM `user_status` с состояниями `active`, `banned`, `deleted`;
   - выполняет `db.AutoMigrate(...)` для моделей пользователей/групп/ролей и связей.
   Плюс: разработка и окружения синхронизируются быстро, без “ручных шагов в БД”.

3. Модели и soft-delete подход уже присутствуют.
   - у пользователей есть поле `Status` и `gorm.DeletedAt`;
   - репозитории реализуют “мягкое удаление” (обновление `status`/`deleted_at` вместо физического удаления).
   Плюс: это удобно для аудита и восстановления данных, и хорошо ложится на дальнейшие требования.

4. Чёткий входной контур приложения.
   - `back/main.go` грузит конфиг из env, поднимает Postgres, запускает миграции и стартует HTTP router (`chi`).
   Плюс: точка входа минимальная и понятная — легче тестировать старт/подключение.

## Запуск локально

1. Поднимите Postgres через Docker:
   - `docker compose up -d postgres`

2. Убедитесь, что env совпадает с `back/.env` (там уже заданы значения по умолчанию):
   - `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`, `DB_SSLMODE`

3. Запустите backend:
   - `cd back`
   - `go run .`

Проверка:
- `GET http://localhost:8080/ping` вернёт `pong`.

## Коротко о подходах (почему их стоит развивать)

Этот каркас специально сделан так, чтобы дальнейшая разработка была “дешёвой”:
- разделение на `domain/repository/usecase/delivery` уменьшает связность;
- миграции в одном месте упрощают поддержку схемы;
- join-таблицы уже выделены как отдельные доменные сущности — значит, дальше будет легко добавлять правила и проверки для ролей в группах/для назначения ролей пользователям.
